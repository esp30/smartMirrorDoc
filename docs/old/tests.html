How we test our system

Each module has 1+ responsibles (available in Log Book) responsible for defining unitary tests for each module, according to the Requirements.
Then the responsibles for the modules decide between them will be integration tests
The whole team decides the acceptance tests and system overall tests, based on the usage scenarios (available in Persona and Scenarios
The end result is the QA report.

Final Results

In addition to the unitary and integration tests done during development, in the final release (Milestone 6), a set of videos were created to showcase that both scenarios (User at the Canteen and Canteen Worker adds a new meal of the day) are fullfilled. See it here.

Results as of Milestone 5

MVP Scenario Result
Scenario 1. User at the Canteen
1A. User Identifies Him/herself upon entrance Working. User is identified using a contactless proximity MIFARE card
1B. Precalculation of the advised portions Working. Predefined user information is used to calculate how much of each food can still be eaten by the user
1C. Measure the weight of the contents in the food plate Working. The weight sensor can correctly measure weight with a precision of 0.01g
1D. Identify what ingredient is being taken of the containers Working. The ultrassonic sensors are working as presence sensors and can automatically identify from which container the food is being taken of
1E. Stand display shows the users advised portion sizes for each available food Validation under way (not fully tested as of 27th May)
Scenario 2. Canteen Worker adds a new meal of the day
2A. Create a new recipe Validation under way (not fully tested as of 27th May)
2B. Create the menus for the next week Validation under way (not fully tested as of 27th May)

Results as of Milestone 4

For this Milestone, even if at a shallow level, the various modules already went through some testing to guarantee their quality.
Module Result
Server API / Database The connection between the API and the database was extensively tested via HTTP Requests, to guarantee it’s functioning correctly. · It was concluded that the functions handle the data correctly, but the response times are above what we were expecting.
CCU Processing Module A nutritionist was consulted to check the if the calculations done by this module were factually correct.
Stand Controller The MQTT broker was stress tested to check if it could handle big volume of usage. · At 50000 messages per millisecond, the module fulfilled our expectations
Stand Displays Interface Validation under way. Only visual validation was done and results shall be presented in Demo 2.
Sensors Validation under way
